# Архитектура Android-приложений

Архитектура Android-приложений построена на принципах, которые обеспечивают гибкость, масштабируемость и поддерживаемость кода. Основой любой Android-системы является жизненный цикл компонентов, который определяет, как приложение взаимодействует с пользователем и системой. Понимание архитектурных компонентов и их правильное использование критически важно для создания качественных приложений. В этой главе мы рассмотрим ключевые компоненты Android и архитектурные паттерны, которые помогают организовывать код эффективно.

## Activity

Activity — это основной строительный блок пользовательского интерфейса в Android. Каждая Activity представляет собой отдельный экран с пользовательским интерфейсом, который может содержать различные элементы управления, такие как кнопки, текстовые поля и списки. Activity управляет своим жизненным циклом, который включает состояния: создание (onCreate), запуск (onStart), возобновление (onResume), пауза (onPause), остановка (onStop) и уничтожение (onDestroy). Жизненный цикл Activity критически важен для правильного управления ресурсами и сохранения состояния приложения.

### Управление жизненным циклом Activity

Жизненный цикл Activity управляется системой Android в зависимости от действий пользователя и системных событий. Когда пользователь открывает приложение, система вызывает метод onCreate, который используется для инициализации компонентов и загрузки макета. Метод onResume вызывается, когда Activity становится видимым и готовым к взаимодействию с пользователем. При переключении на другую Activity вызывается onPause, где необходимо сохранять важные данные. Система может уничтожить Activity для освобождения ресурсов, поэтому важно правильно реализовывать сохранение и восстановление состояния.

## Fragment

Fragments — это повторно используемые компоненты пользовательского интерфейса, которые могут быть объединены в Activity. Каждый Fragment имеет собственный жизненный цикл и может обрабатывать свои собственные события ввода. Использование Fragments позволяет создавать гибкие и адаптивные интерфейсы, особенно для планшетов и устройств с различными размерами экранов. FragmentManager управляет добавлением, удалением и заменой фрагментов в Activity.

### Взаимодействие между Fragments

Взаимодействие между фрагментами должно осуществляться через Activity, которая выступает в роли посредника. Для обмена данными между фрагментами рекомендуется использовать интерфейсы или общую ViewModel. Такой подход обеспечивает слабую связанность компонентов и упрощает тестирование. При разработке важно учитывать, что фрагменты могут быть переиспользованы в различных Activity, поэтому логика взаимодействия должна быть изолирована от конкретной реализации Activity.

## Jetpack Compose

Jetpack Compose — это современный декларативный фреймворк для построения пользовательских интерфейсов на Android, который был представлен Google как альтернатива традиционному XML-разметке. Вместо создания иерархии View с использованием разметки, Compose позволяет описывать интерфейс с помощью Kotlin-кода, используя композируемые функции. Это делает код более читаемым и предсказуемым, так как UI напрямую зависит от состояния приложения.

### Принципы работы Compose

В Jetpack Compose интерфейс описывается как набор композируемых функций, аннотированных `@Composable`. Композируемые функции могут вызывать другие композируемые функции и использовать состояние для обновления интерфейса. Когда состояние меняется, Compose автоматически перерисовывает только те части интерфейса, которые изменились, что обеспечивает высокую производительность. Управление состоянием в Compose строится на принципе однонаправленного потока данных, где события от пользователя поднимаются вверх по иерархии, а состояние спускается вниз.

## Intent

Intent — это механизм обмена сообщениями между компонентами Android. С помощью Intent можно запускать Activity, стартовать сервисы, отправлять широковещательные сообщения и передавать данные между компонентами. Intent бывают явными (explicit) и неявными (implicit). Явные Intent указывают конкретный компонент, который должен быть запущен, например, конкретную Activity. Неявные Intent описывают действие, которое должно быть выполнено, а система определяет подходящий компонент для его выполнения.

### Передача данных с помощью Intent

При передаче данных через Intent можно использовать дополнительные параметры (extras), которые представляют собой пары ключ-значение. Это позволяет передавать простые типы данных, такие как строки, числа и сериализуемые объекты. Для передачи более сложных данных рекомендуется использовать Parcelable или сериализацию в JSON. При отправке данных важно учитывать ограничения на размер данных, передаваемых через Intent, которые обычно составляют около 1 МБ.

## Навигация в Android

Навигация — это фундаментальный аспект пользовательского опыта в мобильных приложениях. Android предоставляет компонент Navigation, который упрощает реализацию навигации между экранами. Навигация в Android может быть реализована с использованием Navigation Graph, который определяет все возможные маршруты в приложении. Компонент Navigation работает с фрагментами и Compose, обеспечивая единый подход к управлению навигацией.

### Компонент Navigation

Компонент Navigation состоит из трех основных частей: Navigation Graph (навигационный граф), NavHost (контейнер для фрагментов или композируемых функций) и NavController (контроллер навигации). Navigation Graph — это XML-ресурс, который определяет все пункты назначения и действия. NavHost используется для отображения текущего пункта назначения, а NavController управляет навигацией между пунктами назначения. Использование компонента Navigation упрощает реализацию сложной навигации и обеспечивает единообразное поведение back-стека.

## Заключение

Архитектура Android-приложений постоянно развивается, предлагая разработчикам новые инструменты и подходы для создания качественных приложений. Современные приложения используют комбинацию классических компонентов (Activity, Fragment) и новых технологий (Jetpack Compose, Navigation). Важно понимать основы жизненного цикла компонентов и уметь применять архитектурные паттерны для организации кода. Это позволяет создавать масштабируемые, тестируемые и поддерживаемые приложения, которые могут развиваться вместе с требованиями бизнеса и технологическими изменениями.
